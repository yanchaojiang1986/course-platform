# 封板流程（SOP）

适用范围：`course-platform` 项目每次执行“封板”操作时，统一按以下步骤执行。

## 1. 确认变更范围
- 执行：`git status --short`
- 明确哪些文件属于本次封板范围，哪些是工具自动文件（如 `CURRENT_TASK.md`、`conversations/*.md`）。

## 2. 执行质量闸门
- 执行：`npm run lint`
- 执行：`npm run build`
- 任一步失败则停止封板，先修复后再继续。

## 3. 更新版本号
- 修改 `package.json` 的 `version`（例如 `1.0.3 -> 1.0.4`）。

## 4. 更新变更记录
- 更新 `CHANGELOG.md`：
- 写明本次新增/变更/修复与校验结果。

## 5. 更新封板记录
- 更新 `SEAL_RECORD.md`：
- 标记当前版本为“当前版本”；
- 记录本次质量校验结果；
- 记录关键文件 SHA-256。

## 6. 复核封板状态
- 再次执行：`npm run lint`、`npm run build`
- 确认封板文档、版本号、代码状态一致。

## 7. 提交封板变更
- 仅提交本次封板相关文件（避免提交会话自动文件）：
- 参考命令：
  - `git add package.json CHANGELOG.md SEAL_RECORD.md <其他本次业务文件>`
  - `git commit -m "chore(release): seal vX.Y.Z"`

## 8. 自动推送 GitHub（新增强制步骤）
- 封板完成后必须立即推送远程仓库，不再等待单独指令。
- 参考命令：
  - `git push origin <当前分支>`
- 推送失败时需要立即反馈失败原因（权限、网络、远程不存在、冲突等），并在修复后重试。

## 9. 封板完成回执
- 输出以下信息：
- 封板版本号；
- 质量闸门结果；
- 提交哈希；
- 推送目标仓库与分支；
- 剩余未提交文件（如有）。

## 10. Zeabur 部署排障协作规则（长期）
- 适用场景：Zeabur 部署失败、服务崩溃、域名接入异常等线上问题。
- 每次排障先查看 Zeabur 右侧 AI 面板输出的结论与建议。
- 将 Zeabur AI 给出的关键报错（构建日志、运行日志、环境变量建议）作为第一优先级线索。
- 执行修复动作前先与 Zeabur AI 结论交叉验证，减少重复排查与误操作。
