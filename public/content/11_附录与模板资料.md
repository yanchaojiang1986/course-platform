## Bug 严重度与优先级分类参考

在缺陷管理系统中设置正确的 Bug 级别非常重要，直接影响开发的修复优先级和上线决策。

| 等级名称 | 别称 | 判别标准 | 例子 |
|---|---|---|---|
| **致命 (Blocker)** | P0 | 阻碍性错误。系统完全无法运行、崩溃死机；或造成严重的资金/数据丢失；导致测试无法继续推进 | 支付界面一点击付款 App 直接闪退；用户数据全部被清空 |
| **严重 (Critical)** | P1 | 核心主干业务流不通，但系统没崩溃；或出现大面积数据计算错误 | 账号能登录，但所有页面显示白屏无响应 |
| **一般 (Major)** | P2 | 次要功能不通，正常容错机制有问题，或存在不复杂的临时规避方案 | 搜索页按"价格排序"没反应，但按"销量排序"正常 |
| **轻微 (Minor/UI)** | P3 | 不影响任何业务逻辑运行，如界面排版错位、文字错误、建议性要求 | 字号应为 14px 但实际是 12px；"登录"被写成"登陆" |

## Bug 排查通用流程表

当发现一个异常行为且网页无确切提示，按以下步骤逐层排查：

**第一步：查前端（F12 Network）**
- 请求发出了吗？
- 请求参数格式是否符合 API 约定？
- 有没有标红的失败请求？

**第二步：查网络响应（状态码）**
- 200：请求成功，问题在于返回数据本身
- 400：传给后端的数据类型/结构不合法被拒绝
- 404：接口路径写错了或后端没有部署此接口
- 500 / 502：后端服务器内部逻辑爆炸或重启

**第三步：查服务器日志（Linux）**
```bash
# 连进服务器，实时滚动查看日志，同时在前端重现问题
tail -f /path/to/error.log
```
通常会刷出 `NullPointerException`、`IndexOutOfBounds` 等详细报错，截图贴给后端开发。

**第四步：查底层数据（MySQL）**
```sql
-- 如果全程 200 但展示不对，查一下数据库里的实际数据
SELECT * FROM table_name WHERE condition = 'xxx';
```
可能是历史脏数据导致的查询异常。

## 常用 HTTP 响应状态码速查

| 状态码 | 含义 | 常见原因 |
|---|---|---|
| 200 | 成功 | 请求正常处理 |
| 201 | 创建成功 | 新资源创建成功（POST） |
| 400 | 请求错误 | 前端传参格式错误、缺少必填字段 |
| 401 | 未授权 | 未登录或 Token 过期 |
| 403 | 禁止访问 | 已登录但权限不足 |
| 404 | 找不到 | 接口路径不存在或资源不存在 |
| 500 | 服务器内部错误 | 后端代码抛出异常 |
| 502 | 网关错误 | 后端服务挂掉或正在重启 |
| 504 | 网关超时 | 后端处理时间过长，请求超时 |

## 常用 SQL 速查

```sql
-- 基础查询
SELECT * FROM users WHERE username = 'test';
SELECT username, email FROM users LIMIT 10;

-- 条件筛选
SELECT * FROM orders WHERE status = 'pending' AND create_time > '2026-01-01';

-- 排序
SELECT * FROM products ORDER BY price DESC;

-- 计数
SELECT COUNT(*) FROM bugs WHERE severity = 'P0';

-- 连表查询
SELECT u.username, o.order_id, o.total_amount
FROM users u
JOIN orders o ON u.user_id = o.user_id
WHERE u.user_id = 123;

-- 更新数据（测试库！慎用！）
UPDATE users SET account_balance = 9999 WHERE username = 'test_user';

-- 删除数据（测试库！慎用！必须加 WHERE 条件！）
DELETE FROM test_logs WHERE create_time < '2026-01-01';
```

## 常用术语表

| 术语 | 含义 |
|---|---|
| PRD | 产品需求文档（Product Requirement Document） |
| TC | 测试用例（Test Case） |
| QA | 质量保证工程师，通常等同于测试工程师（Quality Assurance） |
| RD / Dev | 研发工程师（Developer） |
| PM | 产品经理（Product Manager） |
| UAT | 用户验收测试（User Acceptance Test） |
| Bug / Defect | 软件缺陷 |
| Blocker | 阻塞，指严重到测试无法继续的 Bug |
| Sprint | 敏捷开发中的迭代周期，通常为 2 周 |
| 提测 | 开发完成后把代码移交给测试人员开始测试 |
| 复测 | Bug 修复后重新验证是否真正修好了 |
| 回归测试 | 确保新改动没有影响到原有功能 |
| 冒烟测试 | 提测后快速验证主流程是否可测（如果主流程都跑不通就退回开发） |
| 灰度发布 | 先向一小部分用户发布新版本，观察无问题后全量发布 |
