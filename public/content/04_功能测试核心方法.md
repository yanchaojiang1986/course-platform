## 学习目标

- 你将掌握从需求到测试点、再到测试用例的核心路径

---

## 拿到需求后，测试人员第一步做什么

如果拿到一份需求后直接就开始乱点界面，那么遗漏 Bug 是必然的。所有测试开始之前，都要经历**需求分析**过程。

**显性需求与隐性需求**
- **显性需求**：PRD 上白纸黑字写着的东西。例如："密码长度需为 8-16 位"
- **隐性需求**：PRD 没写，但作为常识和用户体验必须具备的要素。例如：PRD 说可以清空回收站，但没说清空前要不要弹窗确认——作为测试，你需要补充这些隐性需求：不可挽回的数据删除必须有二次确认弹窗！

**需求分析拆解法：业务场景树**
拿到需求后，用脑图工具（如 XMind）将需求拆解：
- **第一层**：大模块定义（如"购物车"）
- **第二层**：子功能定义（如"添加商品"、"结算"、"删除商品"）
- **第三层**：测试点提取（如"添加单件商品"、"添加已存在商品"、"超过库存时添加"）

## 如何提炼测试点

**从页面看测试点**
- 每一个输入框：数据类型、长度限制、必填/选填、特殊字符
- 每一个按钮：正常点击、灰态（不可点击）条件、多次快速点击
- 每一个列表/分页：空数据状态、单条数据、大量数据翻页

**从流程看测试点**
- 正向流程（最顺利的路径走一遍）
- 流程中每个节点的分支（失败、跳转、中断）
- 跨模块的联动（A 操作完，B 模块的数据是否同步更新）

**从异常和边界看测试点**
- 边界值（范围的最小值、最大值、刚好超出的值）
- 异常输入（空值、超长字符串、特殊符号、SQL注入字符）
- 异常状态（网络中断、权限不足、数据被其他人修改过）

## 四大核心用例设计方法

以下是必须掌握的核心用例设计方法，面试必考，工作必用。
<!-- CHECK:q04_3 -->

### 等价类划分法

**核心思想**：在巨大的输入集合中，找一些有代表性的数据来测。代表性的数据有"有效"和"无效"之分，测一个就相当于测了一类。

**场景举例**：系统要求输入人的年龄（1-120 的整数）
- 有效等价类：输入范围内任意合法的，如 20
- 无效等价类：小于1的（如 0、-5），大于120的（如 121），非数字的（如字母'A'、特殊符号'#'），小数的（如 10.5），为空不输入的情况

<!-- CHECK:q04_4 -->

### 边界值分析法

**核心思想**：大量的 Bug 聚集在输入和输出边界上，所以永远要针对边界重点打击。

**场景举例**：系统要求密码长度必须是 8 到 16 位
- 长度 = 7（刚刚不满足最小值，应该报错）
- 长度 = 8（刚刚满足最小值，应该成功）
- 长度 = 16（刚刚满足最大值，应该成功）
- 长度 = 17（刚刚超出最大值，应该报错）

### 错误推测法

**核心思想**：基于测试人员的经验直觉，专门挑选那些容易发生错误的特殊场景。

**场景举例**：测试一个视频播放器
不仅要按常规播放，还要推测：播放中途如果拔掉耳机，声音会不会外放？（好的设计应该自动暂停）播放中途如果有电话打进来，会不会被强制打断而且没有保存进度？
<!-- CHECK:q04_6 -->

### 场景法（业务流程法）

**核心思想**：不再局限于单一输入框，而是从用户真实使用的业务轨迹出发，连成一条线。

**场景举例**：电商购物
- **基本流（最顺利路径）**：浏览商品 → 加入购物车 → 提交订单 → 支付成功 → 等待发货
- **备选流（异常与分支）**：
  - 加入购物车时库存不足
  - 支付时余额不足导致支付失败
  - 支付成功后立刻申请退款

这种方法能保障核心业务流不死机，优先级最高。

## 如何把测试点写成标准测试用例

一个标准的测试用例通常在 Excel 或专业的测试管理平台（如 TestLink、JIRA 等）编写。不论什么工具，必须包含以下核心要素：

| 字段名 | 说明与示例 |
|---|---|
| **用例编号** | 唯一标识符。如：`Login-001` |
| **用例标题** | 一句话概括测试目的。如：`正确账号和密码可以成功登录` |
| **前置条件** | 必须提前满足的状态。如：`当前处于未登录状态，已有注册账号` |
| **执行步骤** | 分步骤清晰描述。如：`1. 进入登录页；2. 输入正确账号X；3. 输入对应密码Y；4. 点击登录按钮` |
| **预期结果** | 核心！如果不写预期就等于没测。如：`1. 页面成功跳转到用户主页；2. 右上角显示用户昵称；3. 数据库产生对应的登录日志记录` |
| **优先级** | P0（最高：阻断性）、P1（核心功能）、P2（边缘/异常功能）、P3（UI细节） |

## 04.5 用例评审——怎么开评审会，怎么应对开发质疑

用例评审会是测试用例正式生效前的"质检环节"。开会的目的不是"念用例"，而是**用集体智慧补全你一个人想不到的场景**。

**评审会要邀请谁**

| 角色 | 评审重点 |
|---|---|
| 产品经理 | 用例是否覆盖了 PRD 的全部需求点、有没有偏离需求 |
| 开发工程师 | 用例是否切到了代码实现的关键路径、有没有冗余 |
| 其他测试同事 | 补充经验上能想到的额外场景 |
| 测试组长 | 把控用例质量、安排执行优先级 |

**开会前你要做的 3 件准备**

1. 提前 24 小时把用例发给参会人，让大家先看
2. 标出你认为有争议或不确定的用例（用红字或备注）
3. 准备好 1 个版本的"测试范围思维导图"，开会先讲全貌再讲细节

**应对开发质疑的话术模板**

- 开发："这个场景不可能发生"
  → 你："我也希望不会发生，但万一发生时，系统应该怎么反应？你的代码里有处理吗？"

- 开发："这是产品需求没写，测了也白测"
  → 你："PRD 没写不代表用户不会这么操作。如果用户这么做了 App 崩了，是产品的锅还是开发的锅？"

- 开发："这个用例太极端，工作量太大"
  → 你："那我们一起评估一下风险——如果不测，万一线上发生，影响的用户量级是多少？"

**评审会一定要产出会议纪要**：哪些用例新增、哪些删除、哪些待确认、责任人、deadline。

## 04.6 移动端测试要额外关注什么

测 Web 和测 App 的差异巨大。App 测试除了功能本身，还要关注**设备、网络、系统、生命周期**这 4 个维度。

**移动端测试专项 Checklist**

<!-- DEMO:04-06 -->

**最容易遗漏的 5 个移动端场景**

1. **安装/卸载/升级**
   - 全新安装 → 数据库初始化是否正确
   - 升级覆盖安装 → 老用户数据是否保留、新表/字段是否迁移成功
   - 卸载重装 → 缓存清理是否彻底、是否要求重新登录

2. **横竖屏切换**
   - 输入到一半切换横屏，已输入内容是否丢失？
   - 视频播放中切横屏，是否自动全屏？

3. **权限管理**
   - 首次开启相机/相册/定位权限的弹窗时机
   - 用户拒绝权限后的降级体验（不能直接闪退）
   - 系统设置里手动关闭权限后再回 App，是否能感知

4. **弱网/断网**
   - 提交订单时网络中断，重连后订单是否重复提交（幂等性）
   - 长列表加载到一半断网，恢复后能否续传
   - 推荐用 Charles / Fiddler 模拟限速：2G/3G/100kbps/0bps

5. **后台/前台切换**
   - App 切到后台 10 分钟再回来，登录状态是否过期
   - 视频播放中切后台，是否暂停（iOS 默认会暂停）
   - 后台时收到 Push，点击 Push 能否跳转到正确页面

**移动端必备的 3 类机型矩阵**

| 类型 | 选机原则 | 示例 |
|---|---|---|
| 主流机型 | 覆盖 80% 用户 | iPhone 13/14/15、华为 Mate 系列、小米数字旗舰 |
| 老机型 | 低端机性能瓶颈 | iPhone SE、千元安卓机（性能弱、内存小） |
| 特殊屏 | 异形屏适配 | 折叠屏、刘海屏、灵动岛、超长屏 |
