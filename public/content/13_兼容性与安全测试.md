## 13.1 Web 端兼容性测试矩阵

不同的浏览器、不同的分辨率、不同的操作系统组合起来是个天文数字。**关键是怎么选有限的测试组合，覆盖 95% 的用户。**

**Web 兼容性测试矩阵设计原则**

**主流浏览器市场份额（2025 年数据，参考）**

| 浏览器 | PC 占比 | 测试优先级 |
|---|---|---|
| Chrome | 65% | P0 必测 |
| Edge | 12% | P1 |
| Safari (macOS) | 10% | P1 |
| Firefox | 5% | P2 |
| 国产（360/QQ等） | 8% | P2（看产品定位） |

**主流分辨率**

| 分辨率 | 占比 | 设备类型 |
|---|---|---|
| 1920×1080 | 25% | 主流台式机/笔记本 |
| 1440×900 | 20% | Mac 笔记本 |
| 1366×768 | 18% | 老款笔记本 |
| 2560×1440 | 10% | 大屏 / 4K 缩放 |
| 移动端 375-414 | 取决于业务 | 手机访问 Web |

**Web 兼容性常见 Bug**

1. **CSS 不兼容**：Safari 不支持某些 CSS 属性（如 `gap` 老版本）
2. **JavaScript API 不兼容**：IE 不支持 `Promise`、老 Safari 不支持 `optional chaining`
3. **字体渲染差异**：Windows 和 macOS 同一字体显示效果不同
4. **图片加载差异**：webp 格式 Safari < 14 不支持
5. **滚动行为差异**：iOS Safari 的橡皮筋滚动 vs Chrome 平滑滚动

**推荐工具**

- **Chrome DevTools 设备模拟**（免费内置）
- **BrowserStack**（付费，云端真机/真浏览器）
- **LambdaTest**（付费，类似 BrowserStack）

## 13.2 移动端兼容性测试矩阵

移动端的碎片化比 PC 严重 10 倍。聪明的做法是建立**机型矩阵**。

**iOS 兼容性矩阵设计**

| 设备 | 系统版本 | 选机理由 |
|---|---|---|
| iPhone SE (第3代) | iOS 17 | 小屏 + 主流系统 |
| iPhone 13 | iOS 17 | 主流刘海屏 |
| iPhone 15 Pro Max | iOS 17 | 灵动岛 + 大屏 |
| iPhone 11 | iOS 15 | 老系统兼容 |
| iPad mini | iPadOS 17 | 平板适配 |

**Android 兼容性矩阵设计**

Android 比 iOS 难，因为厂商 + 系统版本 + 屏幕尺寸三重碎片化。

| 设备 | 系统版本 | 选机理由 |
|---|---|---|
| 小米 14 / Redmi K70 | Android 14 (MIUI 15) | 主流安卓旗舰 |
| 华为 Mate 60 | HarmonyOS 4 | 鸿蒙特殊适配 |
| OPPO Reno / vivo S | Android 13 (ColorOS/OriginOS) | 中端市场 |
| 三星 S23 | Android 14 (One UI) | 海外市场 |
| 老安卓机（Android 10） | 自带千元机 | 性能瓶颈测试 |
| 折叠屏（华为 Mate X / 三星 Z Fold） | 最新 | 异形屏适配 |

**移动端常见兼容性 Bug**

1. **状态栏被遮挡**：刘海屏/灵动岛布局问题
2. **键盘弹起遮盖输入框**
3. **横竖屏切换布局错乱**
4. **深色模式适配缺失**
5. **不同分辨率下图片拉伸/失真**
6. **不同系统返回手势冲突**（iOS 边缘左滑 vs App 内部返回）
7. **第三方 SDK 兼容性**（如微信支付在小米/华为表现不同）

## 13.3 安全测试扫盲——黑盒角度能测什么

功能测试人员不需要会渗透，但要能发现**常识性的安全漏洞**。

**6 大常见安全漏洞（功能测试可发现）**

<!-- DEMO:13-02 -->

### A. 越权漏洞（最高频）

**水平越权**：用户 A 能看 / 改 / 删用户 B 的数据
- 测试方法：拿 A 的 token，把请求里的资源 ID 改成 B 的 → 如果返回 200，就是越权

**垂直越权**：普通用户能调管理员接口
- 测试方法：用普通账号 token 调 `/api/admin/delete-user` → 不该返回 200

### B. SQL 注入

特征字符串：`' OR 1=1--`、`'; DROP TABLE users;--`

**测试方法**：在登录用户名、搜索框等输入框尝试输入这些字符
- ✅ 安全：后端返回 "用户名不存在" 或 400 错误
- ❌ 漏洞：成功登录、或返回数据库报错信息

### C. XSS（跨站脚本）

特征代码：`<script>alert('xss')</script>`

**测试方法**：在评论框、昵称、备注等输入框输入上面代码并提交
- 刷新页面 → 如果弹出 alert 框，说明有 XSS 漏洞
- ✅ 安全：HTML 标签被转义成 `&lt;script&gt;` 文本显示

### D. 敏感信息泄露

**检查清单**：
- [ ] 接口返回是否包含明文密码
- [ ] 手机号/身份证号是否脱敏（138****1234）
- [ ] 服务器报错是否暴露完整堆栈/数据库表结构
- [ ] URL 中是否携带 token（可能被代理日志记录）
- [ ] 前端代码是否硬编码 API key

### E. 越权访问文件

**测试方法**：
- 如果有文件下载接口 `/api/download?file=order_001.pdf`
- 试试 `/api/download?file=../../etc/passwd`（路径穿越）
- 看是否能下载到系统文件

### F. 短信/验证码爆破

**测试方法**：
- 短时间内对同一手机号请求 100 次验证码 → 是否有频次限制
- 同一验证码尝试 100 次 → 是否有错误次数限制
- 验证码过期后能否复用

## 13.5 安全 Bug 怎么提

安全 Bug 比普通 Bug 更敏感，标题和描述要谨慎：

❌ 错误写法：`存在 SQL 注入漏洞，可以拖库`（标题就泄露了攻击方式）

✅ 正确写法：
- 标题：`登录接口存在输入校验缺陷，已私聊详细复现步骤`
- 描述：在 JIRA / 禅道里只写"已通过私信发送给开发负责人"
- 详细步骤通过 IM 单独发送
- **绝不在公开渠道贴出攻击 payload**

**安全 Bug 的处理流程**

1. 发现 → 立刻报告测试组长 + 开发负责人
2. 评估影响范围（是否需要紧急停服）
3. 私下沟通修复方案
4. 修复后**测试人员保留复现脚本作为回归测试用例**
5. **不要在群里讨论**，安全敏感
