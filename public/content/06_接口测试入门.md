## 什么是接口，为什么测试人员要懂接口

**前后端分离的基础认知**
如今的系统都是前后台分离的。前端（手机 APP、PC 网页）就像餐厅服务员，后端（服务器）就像大厨。服务员和大厨之间的"传菜窗口"就是**接口（API）**。

接口是前端和后端之间约定好的通信协议：
- 前端按照约定的格式发送请求（我要一份宫保鸡丁）
- 后端按照约定的格式返回结果（给你，已经炒好了）

**页面正常不代表接口正常**
有时候前端做了"容错处理"，即使接口返回了错误，界面上也不会报错，而是显示一个空白或者默认值——这个时候如果只点界面，就会漏掉 Bug。

**接口异常如何影响前端显示**
- 接口返回错误数据 → 页面展示错误内容（但可能不报错）
- 接口超时 → 页面一直转圈
- 接口挂掉 → 页面白屏或弹出未处理的错误弹窗

## 请求和响应到底是什么

**一次接口调用由两部分组成：请求（Request）和响应（Response）**

**请求的组成**

| 组成部分 | 说明 | 示例 |
|---|---|---|
| **URL** | 接口的地址，告诉服务器要找谁 | `https://api.example.com/users/login` |
| **请求方法** | 描述"要做什么操作" | GET（获取数据）、POST（提交数据）、PUT（更新）、DELETE（删除） |
| **请求头（Header）** | 附加信息，如登录凭证 | `Authorization: Bearer eyJhbGciOi...` |
| **请求体（Body）** | 实际发送的数据，通常是 JSON 格式 | `{"username": "test", "password": "123456"}` |

**响应的组成**

| 组成部分 | 说明 | 示例 |
|---|---|---|
| **状态码（Status Code）** | 表示请求是否成功 | 200（成功）、400（请求错误）、404（找不到）、500（服务器错误） |
| **响应体（Response Body）** | 服务器返回的数据，通常是 JSON 格式 | `{"code": 0, "msg": "登录成功", "data": {...}}` |

## 使用 Chrome DevTools 抓包（F12）

对于网页端的问题，不需要 Postman，直接用浏览器内置工具即可。

**步骤：**
1. 打开浏览器，按 **F12** 打开开发者工具
2. 点击 **Network（网络）** 面板
3. 重新触发刚才的操作（如点击登录按钮）
4. 你会看到出现了一系列的请求数据
5. 找到对应的请求（可以按名称筛选），点进去查看：
   - **Headers**：查看请求方法、URL、请求头
   - **Payload**：查看发给后端的数据
   - **Response**：查看后端返回的数据

**如何判断是前端 Bug 还是后端 Bug？**
- 前端发的数据**有错** → 前端的 Bug
- 前端发的数据**没错**，但后端返回报错 → 后端的 Bug
- 定位 Bug 归属，是高级测试人员区别于小白的重要标志！

## 使用 Postman 发送接口请求

Postman 是接口测试的主流工具，不需要前端界面写好，只要后端写完了接口，你可以立刻利用 Postman 模拟前端发数据。

**发送一个接口请求的步骤：**
1. 打开 Postman，点击 **New → HTTP Request**
2. 选择请求方法（GET / POST 等）
3. 填写接口 URL
4. 如需身份验证，在 **Headers** 中填写 Token
5. 如需发送数据，在 **Body** 中选择 `raw → JSON`，填写请求参数
6. 点击 **Send**，观察下方返回的响应

## 接口测试最基础的验证思路

每次接口测试至少要验证以下四点：

1. **状态码是否正确**：成功通常返回 2xx；失败常见 4xx / 5xx。也有系统约定始终返回 200，再通过业务字段（如 `code`）区分成功失败，需以接口规范为准
2. **返回字段是否正确**：Response Body 里的字段名、字段值是否符合接口文档约定
3. **错误参数时系统是否正确提示**：发送非法参数（空值、超长、错误格式），后端是否返回有意义的错误信息，而不是直接崩溃
4. **接口成功后数据是否联动**：提交成功后，去数据库查一下，数据真的写进去了吗？刷新页面，前端展示的是最新数据吗？

## 复杂系统接口联动：服务方、客户端、中台、上下游怎么测

在企业系统里，接口测试通常不是“单接口校验”，而是“跨系统联动验证”。

**常见链路**
- 客户端系统（调用方） → 业务中台 → 能力中台（账户/支付/风控） → 下游系统（清算/账务/通知）

**你要重点验证的 6 个方面**
1. **字段映射**：字段名、类型、枚举值是否在系统间一致
2. **幂等控制**：同一业务请求重放是否只生效一次
3. **超时重试**：重试是否造成重复处理，是否有去重键
4. **状态一致性**：本系统成功时，下游是否也成功落账
5. **补偿机制**：部分失败时是否能自动回滚或补偿
6. **对账能力**：跨日/批处理后能否识别并修复差异

**测试设计建议**
- 至少设计一条“正常主链路”
- 至少设计一条“下游超时 + 重试”链路
- 至少设计一条“回调重复投递”链路
- 至少设计一条“部分成功 + 补偿”链路

一句话：复杂系统接口测试的本质是“最终一致性验证”。
