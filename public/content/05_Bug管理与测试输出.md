# 模块 05：Bug 管理与测试输出

## 学习目标

- 你将掌握发现问题后的标准化表达能力
- 你将知道测试的工作价值不仅是找 Bug，更是输出质量判断

---

## 什么是一个合格的 Bug

**Bug 的定义**
当你发现软件行为和预期结果不一致时，就需要提交 Bug。不一致的来源有两种：
- **与 PRD 不符**：需求文档明确写了 A，但软件表现为 B
- **与常识不符**：PRD 没写，但违反了基本的用户体验或业务逻辑

**Bug 与需求变更的区别**
- **Bug**：软件的实现与已确认的需求不一致，是开发的问题，必须修复
- **需求变更**：需求本身发生了变化，是产品的决策，走变更流程处理

两者一定要分清。如果你把需求变更当 Bug 提，会浪费开发时间，也损害自己的专业形象。

## 如何写一个开发愿意接的 Bug 单

**Bug 标题怎么写**
使用"在[什么环境/页面]下，进行[什么操作]，发生了[什么错误]"的句式。

- ❌ 坏例子：`登录坏了`
- ✅ 好例子：`在 iOS 端 App 的登录页，输入带空格的用户名点击登录，App 闪退崩溃`

<!-- DEMO:05-02 -->

**完整 Bug 单应该包含什么**

| 字段 | 说明 |
|---|---|
| **标题** | 用句式描述现象，不写模糊词 |
| **重现步骤** | 详细到完全不懂业务的开发人员也能按步骤重现 |
| **实际结果** | 描述"它现在是什么样" |
| **预期结果** | 描述"它本来应该是什么样"（必须有依据，通常引用 PRD） |
| **严重程度** | P0/P1/P2/P3（见附录分类标准） |
| **附件** | 必须附上报错截图或录屏，能极大减少扯皮时间 |
| **环境信息** | 操作系统版本、浏览器版本、App 版本、测试账号 |

## 什么是测试报告

测试报告是每个版本上线前的质量评估文档，是测试人员给出"同意发布 / 不建议发布"结论的正式依据。

**测试报告要写什么**

| 章节 | 内容 |
|---|---|
| **测试范围** | 本次测试了哪些功能模块、哪些版本 |
| **测试结论** | 是否建议上线，理由是什么 |
| **Bug 统计** | 总数、各优先级数量、已关闭数、遗留数 |
| **遗留 Bug 说明** | 遗留了哪些 Bug，为什么可以遗留，风险如何 |
| **风险总结** | 测试未覆盖的范围、已知的潜在风险 |

**为什么测试报告重要**
- 这是测试人员专业性的直接体现
- 上线后出了问题，测试报告是责任边界的重要依据
- 帮助管理层做出有数据支撑的发布决策

## 05.5 JIRA / 禅道实操——提一个 Bug 的完整动作链

不论公司用哪个缺陷管理系统（JIRA、禅道、PingCode、Worktile），提 Bug 的核心动作是一样的。

**提 Bug 的标准 7 步操作**

<!-- DEMO:05-05 -->

**JIRA 和禅道的字段对照表**

| 通用概念 | JIRA 叫法 | 禅道叫法 | 说明 |
|---|---|---|---|
| Bug 标题 | Summary | 标题 | 一句话描述现象 |
| 严重程度 | Priority | 严重程度 | P0–P3 / 致命–建议 |
| 当前负责人 | Assignee | 指派给 | Bug 当前由谁处理 |
| 影响版本 | Affects Version | 影响版本 | 在哪个版本发现 |
| 修复版本 | Fix Version | 解决版本 | 计划在哪个版本修复 |
| 模块归属 | Component | 所属模块 | 属于哪个业务模块 |
| 状态 | Status | 状态 | New/Open/Fixed/Closed 等 |

**新人提单最常踩的 5 个坑**

1. **重现步骤不完整**：少写了前置条件（"已登录""使用 VIP 账号"等）
2. **附件没传**：截图缺失，开发问 3 遍你都说不清
3. **指派错人**：随便指给了一个人，应该指给模块负责人
4. **严重程度乱填**：UI 错位写成 P0，导致开发被半夜叫醒
5. **没写预期结果**：只说"这里不对"，没说"应该是什么样"

**Bug 状态流转的实操要点**

- 开发改为 Fixed 后，测试人员必须 **复测（Retest）**——不能只看代码改没改
- 复测通过才能 **Close**——别"看着像修好了"就关
- 复测没修好要 **Reopen**——并在评论里说明"为什么还是不通过"

## 05.6 测试报告模板精讲——每个字段写什么、怎么下结论

测试报告是你给出"是否同意上线"的正式依据。写不好的测试报告，出了事故责任全在你头上。

**一份完整测试报告的 8 大章节**

| 章节 | 字数 | 写什么 | 示例 |
|---|---|---|---|
| 1. 项目概述 | 100 字 | 本次测试的目标和范围 | 完成 v2.5.0 版本会员中心改版功能测试 |
| 2. 测试范围 | 列表 | 测试了哪些模块、哪些没测 | 已测：登录/注册/会员升级；未测：第三方支付（依赖未到位） |
| 3. 测试环境 | 表格 | 测试用的系统/浏览器/设备 | iOS 14-17 / Android 10-14 / Chrome 120+ |
| 4. 用例执行情况 | 数据 | 总数/通过/失败/阻塞 | 共 153 条，通过 148，失败 3，阻塞 2 |
| 5. Bug 统计 | 数据 + 图表 | 各优先级数量、已关/遗留 | P0×0、P1×2（已关）、P2×8（关 6 留 2）、P3×15（关 10 留 5） |
| 6. 遗留 Bug 说明 | 详述每条 | 哪些 Bug 留下来了，为什么 | BUG-152：搜索排序偶发延迟（P3）——下个版本修复，已与产品确认 |
| 7. 风险与建议 | 列表 | 测试没覆盖的风险点 | 未做压力测试，预计大促可能性能不足；建议运维监控 QPS |
| 8. **测试结论** | 1 句话 | **是否建议上线** | ✅ 建议上线 / ⚠️ 有条件上线（需运维提前监控） / ❌ 不建议上线 |

**写"测试结论"的 3 档标准**

```
✅ 同意上线 = P0=0 + P1 全部关闭 + P2 关闭率 > 80%
⚠️ 有条件上线 = P0=0 + P1 全部关闭，但有 P2 遗留，需明确风险预案
❌ 不建议上线 = P0 仍存在 / P1 未全关 / 主流程跑不通
```

**新人写测试报告的 3 个雷区**

- 🚫 用模糊词："基本正常""差不多可以了"——发生事故时是甩锅口实
- 🚫 不写"未覆盖范围"——出问题时无法证明"这不是我应该测的"
- 🚫 数字凑数：实际只关了 5 个 Bug，写成"已修复主要问题"——签字就要负责
