# 模块 05：Bug 管理与测试输出

## 学习目标

- 你将掌握发现问题后的标准化表达能力
- 你将知道测试的工作价值不仅是找 Bug，更是输出质量判断

---

## 什么是一个合格的 Bug

**Bug 的定义**
当你发现软件行为和预期结果不一致时，就需要提交 Bug。不一致的来源有两种：
- **与 PRD 不符**：需求文档明确写了 A，但软件表现为 B
- **与常识不符**：PRD 没写，但违反了基本的用户体验或业务逻辑

**Bug 与需求变更的区别**
- **Bug**：软件的实现与已确认的需求不一致，是开发的问题，必须修复
- **需求变更**：需求本身发生了变化，是产品的决策，走变更流程处理

两者一定要分清。如果你把需求变更当 Bug 提，会浪费开发时间，也损害自己的专业形象。

## 如何写一个开发愿意接的 Bug 单

**Bug 标题怎么写**
使用"在[什么环境/页面]下，进行[什么操作]，发生了[什么错误]"的句式。

- ❌ 坏例子：`登录坏了`
- ✅ 好例子：`在 iOS 端 App 的登录页，输入带空格的用户名点击登录，App 闪退崩溃`

<!-- DEMO:05-02 -->

**完整 Bug 单应该包含什么**

| 字段 | 说明 |
|---|---|
| **标题** | 用句式描述现象，不写模糊词 |
| **重现步骤** | 详细到完全不懂业务的开发人员也能按步骤重现 |
| **实际结果** | 描述"它现在是什么样" |
| **预期结果** | 描述"它本来应该是什么样"（必须有依据，通常引用 PRD） |
| **严重程度** | P0/P1/P2/P3（见附录分类标准） |
| **附件** | 必须附上报错截图或录屏，能极大减少扯皮时间 |
| **环境信息** | 操作系统版本、浏览器版本、App 版本、测试账号 |

## 什么是测试报告

测试报告是每个版本上线前的质量评估文档，是测试人员给出"同意发布 / 不建议发布"结论的正式依据。

**测试报告要写什么**

| 章节 | 内容 |
|---|---|
| **测试范围** | 本次测试了哪些功能模块、哪些版本 |
| **测试结论** | 是否建议上线，理由是什么 |
| **Bug 统计** | 总数、各优先级数量、已关闭数、遗留数 |
| **遗留 Bug 说明** | 遗留了哪些 Bug，为什么可以遗留，风险如何 |
| **风险总结** | 测试未覆盖的范围、已知的潜在风险 |

**为什么测试报告重要**
- 这是测试人员专业性的直接体现
- 上线后出了问题，测试报告是责任边界的重要依据
- 帮助管理层做出有数据支撑的发布决策
