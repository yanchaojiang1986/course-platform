## 缺陷管理平台：JIRA & 禅道

目前国内外绝大部分公司都会使用项目管理软件来追踪需求和缺陷。

**禅道（Zentao）**
国内开源免费的主流选择，很多中小企业在用。特点是严格遵循产品、测试、开发的角色分离操作，有完整的测试用例管理、Bug 追踪、测试报告功能。

**JIRA**
国际互联网大厂标配，界面高度现代化，配合 Confluence 使用极为强大，支持完全自定义的工作流和看板。

**核心操作：**
- 查看你所在的看板迭代（Sprint），找属于你负责的测试任务
- 点击 "Create Issue"，类型选择 "Bug"，按照 Bug 单规范填写并指派给对应开发人员
- 追踪 Bug 状态变化，在修复后及时复测并关闭

## 浏览器开发者工具（F12）

F12 是测试人员日常最高频使用的工具之一，能帮你在不看代码的前提下判断问题归属。

**常用面板**

| 面板 | 用途 |
|---|---|
| **Network（网络）** | 查看所有接口请求和响应，判断是前端还是后端的问题 |
| **Console（控制台）** | 查看前端 JavaScript 报错信息 |
| **Elements（元素）** | 查看页面 HTML 结构，辅助判断 UI 布局问题 |
| **Application** | 查看 Cookie、LocalStorage 等本地存储数据 |

**Network 面板如何辅助判断问题归属**
1. 在 Network 面板打开的状态下触发问题操作
2. 找到对应的请求（通常标红或状态码异常）
3. 查看 Payload：前端发送的数据有没有错？
4. 查看 Response：后端返回了什么？是错误码还是错误数据？
5. 前端发错了 → 前端 Bug；前端发对了但后端返回错误 → 后端 Bug

## 数据库验证基础：MySQL

测试人员必须会查数据库。因为网页上显示"修改成功"，不代表底层数据库真的存进去了；造测试数据时，自己手写 SQL 插入远比在界面上疯狂点来得快。

**为什么测试要查数据库**
- 验证数据是否真正写入：页面显示≠数据库已存储
- 排查脏数据：某些 Bug 是历史脏数据引起的，不查数据库找不到原因
- 快速造测试数据：准备特定条件的测试场景

**基础必会 SQL**

```sql
-- 查看特定用户的数据
SELECT * FROM users WHERE username = 'zhangsan';

-- 验证某条订单是否存在
SELECT * FROM orders WHERE order_id = '202601010001';

-- 为了测试大额支付，手动给测试账号充值余额
UPDATE users SET account_balance = 9999 WHERE username = 'test_user';

-- 删除无用的测试数据
DELETE FROM test_data WHERE create_time < '2026-01-01';

-- 连表查询：查某用户的所有订单
SELECT u.username, o.order_id, o.total_amount
FROM users u
JOIN orders o ON u.user_id = o.user_id
WHERE u.username = 'zhangsan';
```

**注意**：所有 SQL 操作必须在**测试数据库**中执行，严禁在生产数据库中执行 UPDATE / DELETE！

## Linux 与日志查看基础

绝大部分的测试环境和生产服务器都部署在 Linux 系统上。当系统后台抛了错误，你需要远程连接进去看日志。

**必会五项技能**

```bash
# 1. 实时查看日志（边点页面边看报错）
tail -f /usr/local/tomcat/logs/error.log

# 2. 在日志中搜索关键词（快速过滤报错信息）
grep "Exception" error.log
grep "NullPointerException" error.log

# 3. 切换目录和查看文件列表
cd /usr/local/tomcat/logs
ls -la

# 4. 结束卡死的进程（先温和，再强制）
ps -ef | grep java        # 找到进程号
kill 进程号               # 先尝试正常结束
kill -9 进程号            # 无响应时再强制结束

# 5. 打包解包
tar -czvf backup.tar.gz 文件夹名    # 打包
tar -xzvf backup.tar.gz            # 解包
```

**日志与报错定位的关系**
看到 500 报错后，连进 Linux 主机用 `tail -f` 复现一下问题。日志框通常会刷出 `NullPointerException`、`IndexOutOfBounds` 等详细错误信息——把这段截图贴给后端开发，他会非常感谢你。

## 07.4 Git 基础——测试人员需要懂多少

测试人员**不需要自己写代码提交**，但要能**看懂开发的提交记录**和**自己拉代码切分支**。

**测试人员真正需要的 7 个 Git 操作**

| # | 操作 | 命令 | 用途 |
|---|---|---|---|
| 1 | 拉取代码到本地 | `git clone <url>` | 第一次获取项目 |
| 2 | 更新到最新代码 | `git pull` | 每次测试前同步 |
| 3 | 查看提交记录 | `git log --oneline -20` | 看开发改了什么 |
| 4 | 看某次提交的具体改动 | `git show <commit-id>` | 评估变更范围 |
| 5 | 切换到测试分支 | `git checkout test-v2.5` | 测对应版本 |
| 6 | 查看当前在哪个分支 | `git branch` | 确认环境 |
| 7 | 查看某行代码谁改的 | `git blame <file>` | 定位 Bug 责任人 |

**Git 分支模型：测试人员怎么看**

```
main（生产环境，对应线上）
  ↑
release（预发版本，准备上线）
  ↑
develop（开发主分支，集成所有 feature）
  ↑
feature/xxx（每个需求一个分支）
```

**测试人员的常见场景**

- **拿到开发提测**：开发会告诉你分支名（如 `release/v2.5.0`），你切到该分支、拉代码、部署到测试环境
- **跟踪 Bug 是哪个改动引入的**：用 `git log` 找到最近的提交，挨个 review
- **回归测试**：上线后发现问题，需要知道生产分支和测试分支差了什么 → `git diff main..release`

**测试人员不需要会的**

- ❌ 不需要会 commit、push（不写代码）
- ❌ 不需要会 merge、rebase（不参与代码合并）
- ❌ 不需要会冲突解决（出问题找开发）

**推荐图形化工具**：Sourcetree（免费）或 VSCode 自带 Git 面板，鼠标点点就行，不必死记命令。

## 07.5 测试数据管理——怎么造数据、维护测试账号体系

很多测试人员卡在"没有合适的数据测"。**会造数据是测试人员的核心硬技能**。

**测试数据的 3 种来源**

| 来源 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| **生产数据脱敏导入** | 真实、覆盖广 | 涉及隐私合规风险 | 复现生产 Bug |
| **接口/UI 手动造** | 灵活、可控 | 慢、重复劳动多 | 日常用例执行 |
| **脚本批量生成** | 量大、速度快 | 需要编程能力 | 性能测试、回归测试 |

**测试账号体系怎么维护**

新人测试最常见的灾难：账号混乱——好不容易调出来的边界场景，第二天登录发现账号被同事改过了，全部重测。

**推荐建立一个《测试账号矩阵表》**：

| 账号 | 密码 | 角色 | 状态 | 余额 | VIP等级 | 用途 | 维护人 |
|---|---|---|---|---|---|---|---|
| test_admin_01 | Test@1234 | 超管 | 正常 | 99999 | 钻石 | 后台管理用例 | 张三 |
| test_normal_01 | Test@1234 | 普通用户 | 正常 | 100 | 无 | 日常正向流程 | 李四 |
| test_boundary_01 | Test@1234 | 普通用户 | 正常 | 0 | 无 | 余额边界场景 | 李四 |
| test_frozen_01 | Test@1234 | 普通用户 | 冻结 | 50 | 无 | 异常状态测试 | 王五 |
| test_vip_01 | Test@1234 | 普通用户 | 正常 | 5000 | 黄金 | VIP 权益用例 | 张三 |

**造数据的 5 个实用技巧**

1. **用 SQL 直接批量插入** — 性能用例需要 1000 条订单时最快
2. **接口循环调用** — 用 Postman Runner 跑 100 次注册接口
3. **数据模板化** — 把基础数据存 Excel，每次测试 `INSERT INTO ... SELECT FROM`
4. **脚本生成假数据** — 用固定模板批量生成姓名、手机号、地址等测试数据
5. **保护"种子数据"** — 一些核心账号锁定，不允许任何人修改密码

**测试环境数据库的"3 不要"原则**

- ❌ 不要在生产数据库执行任何写操作（哪怕只是 UPDATE 一行）
- ❌ 不要把生产用户的真实手机号/邮箱导入测试环境（隐私合规）
- ❌ 不要把测试账号密码设为 `123456`（被攻破后能拖库）
