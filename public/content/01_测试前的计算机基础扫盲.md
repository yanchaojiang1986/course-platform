## 电脑、操作系统、软件、浏览器分别是什么

**电脑（计算机）**
就是一台能运行程序的机器。它本身不会做任何事，所有的"聪明"都来自于上面安装的软件。

**操作系统**
操作系统是管理电脑硬件资源、并让其他软件能正常运行的核心程序。常见的有：
- **Windows**：微软出品，绝大多数公司办公电脑使用
- **macOS**：苹果电脑专用，很多互联网公司的研发人员使用
- **Linux**：服务器的主流操作系统，工作中常见以命令行方式操作（也存在带图形界面的发行版）
- **iOS / Android**：手机的操作系统

测试人员最常接触：**Windows**（日常办公）+ **Linux**（查服务器日志）+ **iOS/Android**（移动端测试）。

**软件**
运行在操作系统上的程序。分两大类：
- **客户端（Client）**：安装在你电脑或手机上的程序，如微信桌面版、QQ音乐
- **Web应用**：不需要安装，直接通过浏览器访问的，如京东、飞书网页版

作为测试人员，你会同时测这两类。

**浏览器**
浏览器是一种特殊的客户端软件，专门用来访问网页。常见的有 Chrome（谷歌）、Firefox、Safari（苹果）、Edge（微软）。
- 测试人员最常用 **Chrome**，因为它自带强大的开发者工具（F12），后面会专门讲

**软件安装与卸载**
- Windows：通过安装包（.exe 或 .msi 文件）安装，在控制面板或设置里卸载
- macOS：拖拽 .app 文件到应用程序文件夹安装，拖到废纸篓卸载
- 测试中会频繁安装/卸载，也要测"安装覆盖旧版本是否正常"

## 什么是账号、密码、验证码、缓存、日志

这些是测试工作里最高频出现的概念，必须彻底理解。

**账号（Account）**
唯一标识一个用户的字符串，通常是手机号、邮箱或用户名。测试时你会创建大量测试账号，需要专门维护一份"测试账号列表"。

**密码（Password）**
账号对应的身份验证字符串。测试密码时要关注：
- 长度限制（最小/最大位数）
- 字符限制（是否允许特殊字符）
- 加密传输（密码不应明文出现在网络请求中）
- 错误次数限制（防止暴力破解）

**验证码（Verification Code / CAPTCHA）**
- **短信验证码**：发到手机的 4-6 位数字，有时效限制（通常 5 分钟）
- **图形验证码**：防止机器人批量操作
- 测试验证码时需要关注：过期后能否复用、同一手机号短时间内能获取几次

**缓存（Cache）**
缓存是系统为了加速访问，临时存储的数据副本。常见问题：
- 改了某个配置，页面还是显示旧的——很可能是缓存没清
- 测试时发现的问题，清缓存后消失了——说明这是缓存引发的 Bug
- 清缓存方法：浏览器按 Ctrl+Shift+Delete，手机在设置里清应用缓存

**日志（Log）**
系统运行时自动记录的"操作流水账"。发生异常时，日志是找原因的第一依据。
- **前端日志**：可以在浏览器 F12 的 Console 面板看到
- **后端日志**：存储在服务器上，需要连进去查（后面模块 07 会教）
- 提 Bug 时如果能附上日志截图，开发会非常感谢你

## 什么是前端、后端、服务器、数据库、接口

这是测试人员最需要理解的技术架构，用"点外卖"来类比：

| 技术概念 | 类比 | 说明 |
|---|---|---|
| **前端（Frontend）** | 外卖 APP 界面 | 用户能看见、能操作的部分，用 HTML/CSS/JavaScript 写 |
| **后端（Backend）** | 厨房 + 厨师 | 处理业务逻辑、存取数据，用户看不见，用 Java/Python/Go 等写 |
| **服务器（Server）** | 餐厅建筑本身 | 运行后端程序的机器，24小时不关机 |
| **数据库（Database）** | 仓库/冰箱 | 持久化存储数据的地方，如用户信息、订单记录（常用 MySQL） |
| **接口（API）** | 传菜窗口 | 前端和后端之间约定好的"通信协议"，规定怎么传数据、传什么 |

**为什么测试人员必须理解这个架构？**
因为当出现问题时，你需要判断：
- 是前端显示的问题（界面渲染错误）？
- 是接口的问题（数据传输有误）？
- 是后端逻辑的问题（处理错了）？
- 是数据库的问题（数据本身就错了）？

不懂架构，就只会说"这里有 Bug"，分不清是谁的锅，无法提供有价值的 Bug 报告。

## 测试环境、生产环境、提测、上线是什么

**生产环境（Production）**
就是真实用户在用的那套系统。京东官网、微信 App，这些都运行在生产环境。
- **千万不要在生产环境随意操作！** 一个误操作可能影响真实用户数据

**测试环境（Test / Staging Environment）**
专门供测试人员使用的独立系统，数据是模拟的，操作不影响真实用户。
- 测试人员的所有测试行为必须在测试环境进行
- 测试环境和生产环境可能有配置差异，这也是"测试通过，上线出问题"的原因之一

**开发环境（Development）**
开发人员在自己电脑上运行的本地版本，最不稳定，一般只有开发自己用。

**三个环境的关系：**

<!-- DEMO:01-01 -->

**提测（Submit QA）**
开发人员写完代码、自测通过后，把代码部署到测试环境，正式移交给测试人员开始测试。这是测试执行阶段的起点。

**上线（Release / Deploy）**
测试通过后，把代码部署到生产环境，真实用户就能用到新功能了。

**为什么不同环境结果可能不同？**
- 数据库里的数据不同（生产有真实数据，测试用的是模拟数据）
- 配置参数不同（如支付网关、短信服务商的配置不同）
- 这就是为什么上线后有时会有测试没发现的问题

## 功能测试高频输入知识：数据类型、长度、必填、特殊字符

后续写测试点时，你会不断看到这些词。如果这里不扎实，到了模块 04 会很吃力。

**1）数据类型（Data Type）**
- **字符串（string）**：如用户名 `"alice"`、手机号 `"13800138000"`
- **数字（number）**：如年龄 `18`、价格 `99.5`
- **布尔（boolean）**：如 `true/false`、开关是否开启
- **对象/数组（object/array）**：接口返回里常见嵌套结构
- **null**：有字段但值为空；和“不传这个字段”不是一回事

**2）长度限制（Length Constraint）**
- 常见写法：`4-20 字符`、`8-16 位`
- 测试最少覆盖：`最小值-1`、`最小值`、`最大值`、`最大值+1`
- 例子：用户名 4-20 字符，至少要测 `3/4/20/21`

**3）必填 / 选填（Required / Optional）**
- **必填**：不传字段或传空值，应返回明确错误提示
- **选填**：不传字段应可成功；若传了，仍需校验类型和格式
- 注意区分：`空字符串 ""`、`空格 "   "`、`null`、字段缺失

**4）格式校验（Format Validation）**
- 手机号：位数是否正确、是否只允许数字
- 邮箱：是否包含 `@` 和合法域名
- 日期：是否符合格式（如 `YYYY-MM-DD`）

**5）特殊字符（Special Characters）**
- 常见字符：`' " < > / \\ % _ & + ? =`
- 风险：XSS、SQL 注入、转义错误、截断显示
- 最少要测：中英文、空格、emoji、特殊符号混输

**6）常见误区**
- “页面提示成功”不代表数据正确写入
- “能提交”不代表校验逻辑正确（可能漏校验）
- “接口200”不代表业务成功（可能 `code != 0/200`）

## 接口测试前置知识：参数、状态码、业务码、JSON结构

模块 06 会深入讲接口测试，这里先把最常见概念打平。

**1）接口参数是什么**
- 请求由 `URL + Method + Headers + Body` 组成
- 你要关注每个参数的：`类型`、`是否必填`、`长度`、`取值范围`

**2）状态码（HTTP Status）与业务码（Business Code）**
- 状态码：HTTP 层是否成功（如 `200/400/401/500`）
- 业务码：业务层结果（如 `code: 0` 或 `code: 1001`）
- 现实中两种风格都存在：
  - 失败返回 4xx/5xx
  - 全部返回 200，通过业务码区分成功失败
- 结论：以接口文档约定为准

**3）JSON 结构基础**
- 对象：`{ "name": "alice" }`
- 数组：`[{...}, {...}]`
- 嵌套：`data.user.profile.nickname`
- 测试时要检查：字段是否存在、类型是否正确、值是否符合预期

**4）接口高频错误类型**
- 缺字段：必填参数没传
- 类型错：该传 number 却传 string
- 越界：长度超限、数值超范围
- 枚举非法：只允许 `male/female` 却传 `unknown`
- 鉴权失败：token 缺失或过期

**5）最小验证清单（记住这 5 条）**
1. 状态码是否符合约定
2. 业务码和错误信息是否符合约定
3. 返回字段是否齐全，字段类型是否正确
4. 异常参数是否被正确拦截
5. 成功后数据是否真实落库并回显正确

## 后续模块高频知识点速查（先混个脸熟）

下面这些词会反复出现，提前认识会显著减少掉队概率。

| 术语 | 你现在先记住什么 |
|---|---|
| 等价类（Equivalence Class） | 用少量代表数据覆盖一大类输入 |
| 边界值（Boundary Value） | Bug 最爱出现在边界附近 |
| 场景法（Scenario-based Testing） | 按真实业务流程串起来测 |
| 幂等（Idempotency） | 同一请求重复提交，结果应可控 |
| 分页（Pagination） | 页码、页大小、总数、空页要验证 |
| 排序（Sorting） | 升序/降序是否稳定，字段是否正确 |
| 大小写敏感（Case Sensitivity） | `abc` 和 `ABC` 是否被当成同一个值 |
| 空值处理（Null/Empty Handling） | `null`、空字符串、空格、缺字段要分开测 |

**给零基础同学的执行建议（非常重要）**
- 每次做题时都问自己：这是在测“类型 / 长度 / 必填 / 格式 / 特殊字符”中的哪一个？
- 每次提 Bug 都补一句：`实际结果 vs 预期结果`，不要只说“有问题”
- 不懂术语先查附录，再回到当前题目继续做，不要卡住半天

## 复杂系统测试认知：你测的通常不是“一个页面”，而是“一个业务域”

很多新人会误以为测试=测登录页、注册页。真实工作里，你更常见的是长期维护一个庞大系统中的某个业务域。

**典型场景**
- 银行网银系统：账户管理、转账、额度、风控、对账
- 商业汇票系统：出票、背书、贴现、到期兑付、状态流转
- 电商平台：商品、库存、价格、订单、支付、售后

**你要建立的“系统分层”视角**
1. **系统层**：这是哪个系统（网银、票据、电商）？
2. **子系统层**：用户中心、订单中心、支付中心、风控中心
3. **模块层**：查询、创建、审核、撤销、状态变更
4. **规则层**：权限规则、金额规则、时效规则、状态机规则

**为什么这很重要**
- 复杂系统 Bug 很少是“按钮点不动”，更多是“规则冲突”和“状态错乱”
- 你如果只测单页面，很容易漏掉跨模块影响
- 高质量测试必须覆盖“主流程 + 分支流程 + 异常流程 + 回滚流程”

**复杂系统测试最小清单**
- 模块边界是否清晰（输入/输出是什么）
- 状态流转是否闭环（从A到B能否回退、撤销、重提）
- 权限与角色是否隔离（谁能看、谁能改、谁能审）
- 数据一致性是否可靠（页面、接口、数据库、报表一致）

## 外部系统交互认知：服务方、客户端、中台、上下游

现实里的系统几乎都不是孤岛。你测的系统通常要和多个外部系统交互。

**常见角色**
- **调用方 / 客户端（Client）**：发起请求的一方
- **服务方（Provider）**：提供接口的一方
- **中台（Middle Platform）**：承接通用能力（用户、支付、风控、消息），给多个业务系统复用
- **上游系统**：给你输入数据的系统
- **下游系统**：接收你输出数据的系统

**高频交互模式**
- **同步调用**：请求后马上返回结果（如实时查询余额）
- **异步调用**：先受理，稍后通过消息/回调通知（如清算、批处理）
- **文件交互**：按批次导入导出（如对账文件）

**跨系统最容易出的问题**
- 字段映射错误（A系统叫 `userId`，B系统叫 `customerNo`）
- 时间和时区不一致（当天交易跨日对不上）
- 幂等失效（重复请求导致重复扣款/重复记账）
- 超时重试导致脏数据（第一次成功但回包丢失，第二次又成功）
- 对账不平（你系统成功，下游系统失败，状态不一致）

**你至少要会测的 5 类联动场景**
1. 上游参数合法但下游拒绝
2. 下游超时或不可用（降级/重试/告警是否正确）
3. 回调重复投递（是否幂等）
4. 部分成功部分失败（是否有补偿机制）
5. T+1 对账差异（能否定位并修复）
