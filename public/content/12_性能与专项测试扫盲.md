## 12.1 性能测试到底在测什么

很多人对性能测试的误解："就是看看 App 卡不卡。"

**性能测试 = 在特定压力下，系统的响应能力和稳定性表现。**

**4 个必须看懂的核心指标**

<!-- DEMO:12-01 -->

| 指标 | 全称 | 含义 | 健康值参考 |
|---|---|---|---|
| **TPS** | Transactions Per Second | 每秒处理的事务数 | 列表查询 > 500、下单 > 100 |
| **响应时间** | Response Time | 单次请求从发出到收到响应的耗时 | 95% 的请求 < 1 秒 |
| **并发用户数** | Concurrent Users | 同时操作系统的人数 | 取决于业务量级 |
| **错误率** | Error Rate | 失败请求 / 总请求 | < 0.1% |

**4 种最常见的性能测试场景**

- **负载测试**：逐步加压，观察系统在不同负载下的表现，找到瓶颈点
- **压力测试**：持续超出系统能承受的负载，验证降级和恢复能力
- **并发测试**：多用户同时操作同一资源（如秒杀同一商品）
- **稳定性测试**：长时间（如 24 小时）中等负载运行，找内存泄漏

## 12.2 功能测试人员要看懂性能报告

性能测试报告里 8 个关键字段，看懂它你就能参与发布决策。

**性能报告标准字段对照表**

| 字段 | 怎么看 | 异常信号 |
|---|---|---|
| 总请求数 | 越大越准确 | < 1000 时数据不可信 |
| 平均响应时间 | 整体快慢 | > 3 秒就要警惕 |
| **95% 响应时间** | 95% 的请求都比这个快 | 比平均值大很多 → 长尾问题严重 |
| **99% 响应时间** | 极端慢请求的边界 | > 5 秒 → 有少量请求卡死 |
| TPS | 系统吞吐能力 | 持续下降 → 性能拐点 |
| 错误率 | 失败请求占比 | > 1% 即不可上线 |
| CPU 使用率 | 服务器 CPU 压力 | 持续 > 80% 危险 |
| 内存使用率 | 服务器内存压力 | 持续上涨不下降 → 内存泄漏 |

**为什么要看 P95/P99 而不只是平均值**

如果 100 个请求里 99 个 100ms，1 个 30 秒：
- 平均响应时间 = (99×0.1 + 30) / 100 = **0.4 秒**（看起来不错）
- P99 响应时间 = **30 秒**（真相暴露）

**性能 Bug 怎么描述**

❌ 坏例子：`系统很卡`
✅ 好例子：`在 100 并发下，订单提交接口 P95 响应时间从 800ms 增加到 4.5 秒，错误率从 0.05% 增加到 2.3%`

## 12.3 弱网与断网测试——移动端必测

移动用户的真实网络远比开发桌面环境恶劣。**80% 的线上奇怪 Bug 都和弱网有关。**

**弱网测试要覆盖的 5 个等级**

| 等级 | 模拟带宽 | 模拟丢包率 | 对应真实场景 |
|---|---|---|---|
| 4G | 10 Mbps | 0% | 城市正常网络 |
| 3G | 1 Mbps | 1% | 老城区/电梯 |
| 2G | 200 kbps | 5% | 高铁/地下车库 |
| 极弱 | 50 kbps | 10% | 极偏远地区 |
| 断网 | 0 | 100% | 飞行模式 |

**弱网下必测的 8 个场景**

1. App 启动是否会一直转圈不报错
2. 列表加载到一半断网，是否有"加载失败请重试"
3. 提交订单时网络中断，订单是否会"卡在中间状态"
4. 上传图片/视频中断，是否能续传或友好提示
5. 长视频播放断流，是否自动降码率/暂停
6. 已登录用户突然没网，是否能继续看缓存内容
7. 弱网恢复后，未发出的请求是否会重新发送
8. 同一个操作快速重试 3 次，是否会创建 3 条数据（幂等性）

**怎么模拟弱网**

- **Chrome 开发者工具** → Network 面板 → 选择 Throttling 预设（Slow 3G、Fast 3G）
- **Charles** → Proxy → Throttle Settings → 自定义带宽和丢包率
- **iOS Network Link Conditioner**（开发者工具，免费）
- **小米/华为开发者选项**里直接有弱网模式

## 12.4 并发与幂等性测试

并发是测试人员最容易漏的领域，也是出大事故最多的领域。

**什么是幂等性**

> 同一个操作执行 1 次和执行 N 次，结果应该相同。

**真实事故举例：**
> 用户点击"支付"按钮，因网络延迟没响应，连点了 3 次。如果接口没有幂等保护，扣了 3 次钱、创建了 3 个订单。

**测试人员怎么验证幂等性**

| 方法 | 操作 | 验证点 |
|---|---|---|
| UI 快速连点 | 在 1 秒内点击同一按钮 3 次 | 数据库只产生 1 条记录 |
| Postman 并发 | 用 Runner 同时发 10 次同一请求 | 9 个被拒绝或合并为 1 个 |
| JMeter 压测 | 50 并发提交相同订单 | 库存正确扣减、不超卖 |

**典型需要幂等的接口**

- 创建订单
- 支付/扣款
- 注册/绑定手机号
- 点赞/收藏
- 发送短信验证码

**幂等性失败的 4 个典型现象**

1. 创建了重复订单
2. 库存超卖（卖了 100 件库存的，实际下单 150 单）
3. 重复扣款
4. 同一行为产生多条消息推送

## 12.5 性能测试和功能测试人员的协作

**功能测试在性能测试中的 3 个职责**

1. **提供准确的业务场景**：告诉性能测试同学"哪些接口是核心"
2. **提供测试数据**：批量造账号、商品、库存
3. **结果验证**：跑完性能测试后，检查数据是否一致

**功能测试和性能测试的边界**

| 项目 | 功能测试 | 性能测试 |
|---|---|---|
| 测试目标 | 功能正确性 | 系统承压能力 |
| 数据量 | 单条数据 | 大批量数据 |
| 用户数 | 单用户操作 | 高并发模拟 |
| 关注指标 | Bug 数量 | TPS / 响应时间 |
| 工具 | Postman / 手工 | JMeter / LoadRunner |
